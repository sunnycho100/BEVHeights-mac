# BEVHeight Mac CPU Inference — Plan (3/18/2026)

## Goal

Run BEVHeight inference (evaluation only) on Mac without CUDA.  
NOT training. NOT fine-tuning. Just: "load model → run on small dataset → get outputs."

---

## What Blocks Us

Three things prevent the code from running on Mac as-is:

1. **Custom CUDA voxel pooling kernel** — a compiled `.cu` kernel that won't build without NVIDIA GPU
2. **Hardcoded `.cuda()` calls** — tensors forced onto GPU in several files
3. **PyTorch Lightning trainer config** — defaults to `accelerator='ddp'` and 8 GPUs

All are fixable with ~30 lines of changes across 3–4 files.

---

## Files to Modify

| File | Change | Why |
|------|--------|-----|
| `ops/voxel_pooling/voxel_pooling.py` | Add pure PyTorch `scatter_add_` fallback (~15 lines) | Replace CUDA kernel with equivalent Python logic |
| `ops/voxel_pooling/__init__.py` | Wrap import in try/except | Gracefully handle missing CUDA `.so` |
| `layers/backbones/lss_fpn.py` (line 524) | `self.voxel_num.cuda()` → `self.voxel_num` | Buffer auto-moves with model via `.to(device)` |
| `exps/dair-v2x/bev_height_lss_r50_864_1536_128x128_102.py` | Change trainer defaults (accelerator, gpus) | Enable single-device CPU execution |

## Phase 1: Replace CUDA Voxel Pooling with Pure PyTorch (Critical)

The CUDA kernel (`ops/voxel_pooling/src/voxel_pooling_forward_cuda.cu`) does one thing:

> For each point in `geom_xyz [B, N, 3]`, scatter-add its feature vector from `input_features [B, N, C]` into an output BEV grid `[B, voxel_y, voxel_x, C]` at the given integer coordinates.

This is a standard `torch.scatter_add_` operation — ~15 lines of pure Python. We add a conditional: if the CUDA extension loads, use it; otherwise use the PyTorch fallback.

Also update `__init__.py` to catch `ImportError` when the CUDA `.so` doesn't exist.

## Phase 2: Patch Hardcoded `.cuda()` Calls

- **`layers/backbones/lss_fpn.py` line 524**: `self.voxel_num.cuda()` → `self.voxel_num`  
  It's a registered buffer — auto-moves with the model.

- **Experiment config** (`eval_step` and `training_step`): The `.cuda()` calls are already behind `if torch.cuda.is_available():` guards. These work as-is on CPU — no change needed.

- **`layers/heads/bev_height_head.py`**: Has 6 hardcoded `device='cuda'` allocations, but ALL are inside `get_targets()` which is **only called during training**. Not needed for inference — skip.

## Phase 3: Fix PyTorch Lightning Trainer for CPU

In the experiment config's `run_cli()`:
- `accelerator='ddp'` → `None` (DDP requires multiple GPUs)
- `--gpus` default → `0` for CPU

Run command becomes:
```bash
python exps/dair-v2x/bev_height_lss_r50_864_1536_128x128_102.py \
    --ckpt_path ./checkpoints -e -b 1 --gpus 0
```

## Phase 4: Checkpoint Loading

- `BEVHeight_R50_128_102.4_65.48_49_epochs.ckpt` is already in the repo root
- PyTorch Lightning's `trainer.test(ckpt_path=...)` uses `torch.load` — with `gpus=0`, it should auto-apply `map_location='cpu'`
- If not, one-line patch to force `map_location='cpu'`
